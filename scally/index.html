Why use a CSS framework?

Because CSS sucks!

  • It's old, really old.
  • It operates in a global namespace.
  • It's very, very loose.
  • Source order is critical.
  • Specificity negates the above.
18 years old
No encapsulation
Very leaky
CSS's main feature; the cascade is it's worst
Can lead to a mess.

Our CSS best practices have been killing us

  • Classitis!
  • Never add a non-semantic element.
  • Or, a non-semantic class.
  • Use descendant selectors exclusively.
  • Sites need to look exactly the same in every browser.

http://www.stubbornella.org/content/2011/04/28/our-best-practices-are-killing-us/

Nicole Sullivan 4 years ago.
Style almost everything with classes.
No ID's.
Pride myself on only using 1 class selector.
Doesn't work in the real world.

Writing CSS is easy, architecting it isn’t.

Using no set system things will go downhill fast.

OOCSS to the rescue!

Separation of structure from skin
Separation of containers and content

As with any object-based coding method, the purpose of OOCSS is to encourage code reuse and, ultimately, faster and more efficient stylesheets that are easier to add to and maintain.

Structural CSS is separated from look and feel.
Styles are not dependent on any containing element. Can be reused anywhere in the document, regardless of structural context.

What is Scally?

Bits and pieces to construct UI.

Lego™ is the best analogy for Scally.
Building blocks to build UI.

Scally is a collection of tiny, composable, configurable, interdependent modules that you can piece together how you want, when you want — giving you a solid, scalable CSS architecture.

Provides the structural building blocks to build out nearly anything in a modular way.

Key features

  • Highly modular, following OOCSS principles.
  • Lightweight: 37kb minified, 14kb minified and gzipped.
  • Responsive ready.
  • Unopinionated about design.
  • Uses Sass and BEM.
That's everything, pick and chose what you want.
Everything is fluid.
Mobile first.
Concept of changing UI at breakpoints.
Go non-responsive if you wish.
Zero design treatments.
Everything is configurable.
BEM a front-end naming methodology.

What it's good at

DRYness / Abstraction / Efficiency

  • Providing a solid CSS architecture.
  • Being unopinionated and highly configurable.
  • Being performant (light-weight).
  • Enabling rapid UI development.
  • Ensuring UI's are consistent and less prone to breakages.
  • Conventions and Guidelines.
Everything has it's place.
Highly maintainable.
Build new features quick.
Helps with QA.
Prevents bespoke styles leaking in.
Make far-reaching changes to your UI with simple modifications.
Confidence in your changes because edits to a layout module only ever alter one responsibility.
BEM.
Solid guidelines.

What Scally isn't

An all-singing all-dancing UI Toolkit à la Twitter Bootstrap or Foundation.

Scally doesn't try to be all things to all men, it does one job and one job well, then gets out of your way.

Doesn't focus on providing product specific components.
Provides structural building blocks to build out nearly anything in a modular way.
Architected as a framework for large and small UI builds with production quality coding standards in mind.

How it came about?

Scally was created as an open source CSS framework during my time at Westfield Labs.

It extended and nicely packaged up the CSS framework that I'd been developing and using over many years as a UI developer.

The Structure

Core / Layout / Components / Utilities

Each section acts as a layer building on top of each other.
Core is the foundation: sensible styles, settings, reset.
Layout are the main bits of Lego®.
Components are the discrete custom elements of a UI that enclose specific semantics and styling. Everything is a component.
Utilities are the workhorses.
They have a very narrow scope and may end up being used frequently.
Follow the open/close principle.

Scally: Core / Layout / Components / Utilities

Scally is the Scaffolding, the foundation.
CM product styling sits on top.
Applies the brand and design, and product specific components.
Can be super opinionated.

Settings

You can completely modify how Scally works without ever having to alter the framework itself. Simply redeclare the settings you wish to change above the relevant @import in the master style sheet e.g.

$font-size: 14;
$font-family-base: Arial, sans-serif;
@import "bower_components/scally/core/settings/typography";
$colour-brand: #2aa198;
@import "bower_components/scally/core/settings/colours";
$text-input-and-select-text-color: $colour-text-base;
@import "bower_components/scally/core/base/forms";
Everything is configurable.
Controlled by Sass variables.

Apply at breakpoints

When building responsive UI's it is a really common requirement to apply a style or a set of styles at a specific viewport.

Each Scally utility and layout module comes with this ability.

So if we wanted to hide a Call Us button on larger viewports we would use the Display utility to achieve this, applying these steps…

  1. Turn the feature on by changing the toggle setting to true which will output (compile) the utility in a media query like so: 
    @media (min-width: 40.0625em) {
  .u-hide-from-lap {display: none;}
}
  2. By default the lap breakpoint is used but you can change this to another breakpoint or add more breakpoints above the relevant @import in the master stylesheet: 
    $u-display-breakpoints: (lap, 999);
@import "bower_components/scally/utilities/u-display";
    This will output (compile): 
    @media (min-width: 40.0625em) {
  .u-hide-from-lap {display: none;}
}
@media (min-width: 56.3125em) {
  .u-hide-from-999 {display: none;}
}
Another powerful application of this is defining different widths at viewports.
Means we can have a 6 col grid at large vp's, 3 at medium, and none at small vp's.

Scally's breakpoints

$breakpoints: (

  "palm":         640,

  "lap":          641,
  "lap-large":    901,

  "desk":         1200,
  "desk-large":   1366

) !default;
